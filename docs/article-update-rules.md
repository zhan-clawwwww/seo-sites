# 文章更新规则（前瞻预测 + 当日资讯 · 深度长文）

> **【日常更新以何为准】** 自 **2026-04** 起，**SEO 编写标准、活跃专栏范围、每日篇数** 以 **[`SEO-编写标准与更新要求.md`](./SEO-编写标准与更新要求.md)** 为**执行依据**（当前：**仅 `ai` / `web3` / `tesla` / `apple` 四专栏；每专栏每日 3 篇**）。本文第 1.1 节表内 *8 专栏* 描述仍反映**仓库目录与导航**，不代表日常必须八站齐更。

> **全站内容语言：英文（English only）**。Meta、作者说明与 CI 可保留中文，但**所有已发布 `sites/*/posts/*.md` 的正文、标题、摘要、分类与标签均为英文。**

> 适用于仓库 `sites/` 下各站点内容生产，目标：**满足 Google  helpful content / 时效与深度信号**，以**可验证的“今日信息”**为地基，以**可解释的前瞻预测**为差异化。

## 2. 总目标：两条主线必须同时出现

| 主线 | 对读者的价值 | 对 Google 的语义 |
|------|----------------|------------------|
| **A. 今日最新（Freshness）** | 知道「此刻业界发生了什么」 | 强时效、可引用来源、与查询意图对齐 |
| **B. 前瞻预测（Predictive / Forward-looking）** | 知道「可能往哪走、自己该怎么准备」 | 信息增益、长查询与深度子话题覆盖 |

**禁止**只写空泛年景展望而无近期事实锚点；**也禁止**只堆新闻简讯而无分析框架与预测段落。

## 3. 工作流：先搜再写，先汇总再立论

### 3.1 检索阶段（每专栏必做）

1. **先搜索**与专栏主题强相关的 **近 7～14 天** 英文一手信息：主流媒体科技版、官方博客、监管/安全公告、厂商 release notes、财报电话会摘要、重要会议（如 GTC、WWDC 若临近）等。
2. 至少 **3 个独立信源**交叉核对同一关键事实；冲突处必须在文中写清「各派说法」而非强行定论。
3. 记录**可引用的信息要素**：机构名、人名/职务（若公开）、**日期**、**原文标题级要点**；写作时转述并归纳，**禁止大段机翻式抄袭**。

### 3.2 汇总与「深度」从哪里来

在事实层之上，每篇须包含下列 **不可删减块**（顺序可微调，标题可本地化）：

1. **导言**（用 1 屏说清：今日为何值得读、本文预测回答什么问题）。  
2. **今日/本周要点综述**（分点列出已核实动态，**每条尽量带来源性质说明**，如 “per vendor advisory” / “as reported by …”）。  
3. **多源交叉解读**（把多条新闻串成一条因果链或矛盾点：技术、商业、政策、用户侧各挑至少一层）。  
4. **前瞻预测**（**明确标注**为 *forecast / scenario*，非事实承诺）：  
   - 至少给出 **2 个时间尺度**：**近季（0–3 个月）** 与 **近一年（3–12 个月）**；  
   - 每尺度至少 **2 个可证伪的预测**（即未来可对照复盘）；  
   - 每预测写清 **关键假设**与 **会推翻该假设的观测信号**（什么新闻一出就说明猜错）。  
5. **对读者/业界的行动含义**（可执行：检查清单、决策树、或「若你是 X 角色应关注 Y」）。  
6. **风险、误区与反证**（YMYL 类主题尤须写：监管、投资、安全、健康相关表述的边界）。  
7. **小结 + 可检索结语段**（自然包含 3～6 个专栏核心长尾词，**禁止**关键词堆砌式列表）。

> **深度**在本规则中的操作定义：同一主题下，读者能同时获得 **(a) 事实层时间线 (b) 机制层解释 (c) 可检验的预测与边界条件**，并能在其他转载稿里**抄不走的**是你对矛盾与折衷的论证。

## 4. 篇幅与质量底线

- **正文字数**（**全站英文**）：**不少于 3000 个英文单词**（`wc -w` 或编辑器以 **body 为主，不含 frontmatter**）。**不再发布中文正稿。**
- **禁止**用重复句式、同义改写同一事实凑字数。  
- **信息密度检查**：长段落中每 **3～4 个自然段** 应出现至少 **1 个** 非复读性新信息（新数据、新机制、新比较视角或新预测）。

---

## 5. SEO 与 Google 侧注意事项（不替代质量）

- **标题**：包含专栏核心词 + 时间维度或「2026 场景」，避免纯点击 bait。  
- **元描述/摘要**：`excerpt` 写清 **今日锚点 + 预测价值**（1～2 句，非全抄首段）。  
- **结构化**：H2/H3 层级清晰，列表与表格只用于**真的方便理解**的归纳。  
- **时间字段**：`pubDate` 为**真实发布日**；若文内复盘历史事件，用正文日期，勿混用。  
- **E-E-A-T 友好**：涉及安全/金融/健康时，区分「事实 / 行业惯例 / 作者推断」。  
- **内链**（有则更好）：在相关旧文存在时，自然链到本站同站点的历史深度稿。

## 6. Frontmatter 规范（与现稿对齐）

新文章须包含与现有 `posts` 中一致的元数据，至少：

```yaml
---
title: "英文标题：含领域词 + 时间或场景"
pubDate: YYYY-MM-DD
author: "WordOK Tech Publications"   # 或项目约定笔名
category: "与专栏主题一致的英文类名"  # 如 Artificial Intelligence, VPN, Apple …
tags: ["5～10 个具体长尾或实体词", "含年份 2026 若适用"]
excerpt: "1～2 句，概括今日动因 + 预测看点"
---
```

> 若后续站点引入 `topic` 等额外字段，以 **当时仓库内多数文章** 与 `new-column-planning.md` 为准，本规则不替代代码中的 schema。
