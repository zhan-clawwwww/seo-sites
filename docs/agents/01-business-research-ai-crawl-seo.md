# 业务研究交接：面向检索与 AI 抓取的内容与版式

> **角色**：业务研究员  
> **受众**：产品经理、工程、运营  
> **结论摘要**：「给人看的站」与「给模型与抓取器用的站」并不矛盾，但**信息架构**需要额外一层：**可抽取、可引用、可去重**。

---

## 1. 背景：为什么要在意「AI 抓取」

- 生成式搜索与助手（Google AI Overviews、Perplexity、ChatGPT 浏览、Bing Copilot 等）会组合 **网页正文、结构化数据、站点级说明文件**。
- 站点若只有重 JS、弱标题层级、无稳定摘要，模型侧更容易 **漏抽、误引或合并错误**。
- 目标不是「欺骗爬虫」，而是 **提高机器可读性与引用质量**，与人类可读性一致：清晰、可验证、有日期与边界。

## 2. 内容与版式：经实践归纳的清单

### 2.1 版式与 HTML 语义

| 信号 | 说明 |
|------|------|
| **单一 H1 + 递进 H2/H3** | 便于分段索引与摘要；避免多个 H1。 |
| **首屏可抽取摘要** | 开篇或独立块给出 3～6 句「结论先行」，便于被当作 answer span。 |
| **列表与表格** | 规格对比、步骤、术语表用列表/表格，优于长段散文。 |
| **稳定锚点** | 重要小节带可预测 `id`，便于 `#fragment` 引用（若主题需要）。 |
| **少依赖首屏 JS 才出现正文** | 静态 HTML 内可见正文对抓取更友好（本站 Astro SSG 已满足）。 |

### 2.2 元数据与结构化数据

| 信号 | 说明 |
|------|------|
| **Article / WebPage JSON-LD** | 标题、描述、日期、作者、发布者。 |
| **FAQPage**（若确有问答） | 问答成对、答案自洽，避免空洞重复。 |
| **BreadcrumbList** | 强化栏目层级，利于理解与导航类查询。 |
| **SpeakableSpecification**（可选） | 标明适合朗读/摘要的 CSS 选择器（需在页面真实存在对应节点）。 |

### 2.3 站点级「机器说明」

| 机制 | 说明 |
|------|------|
| **`/llms.txt`（约定俗成）** | 根路径文本文件，说明站点用途、推荐入口、政策边界；多数爬虫可发现。 |
| **RSS / Sitemap** | _feed 与 sitemap 仍是批量发现 URL 的主通道。 |
| **纯文本镜像（可选）** | 对长文提供 `text/plain` 版本，降低 HTML 噪声，利于 RAG 与离线语料处理。 |

### 2.4 内容体裁：对模型「好用」的正文特征

- **显式实体**：人名、产品名、标准名、版本号在首次出现时写全。
- **时间与适用范围**：写清「截至某日」「某地区监管不同」等，减少幻觉外推。
- **可证伪陈述**：预测类用「假设 / 观测信号 / 若失败则…」结构。
- **避免同义堆叠凑字**：重复段落对模型与人类都无增益，且可能触发质量类算法。

### 2.5 「表情/符号」类内容

- Emoji、ZWJ 组合旗、肤色修饰符等，对 **NLP 分词、多模态对齐、词典建设** 有真实需求。
- 版式建议：每个符号给出 **Unicode 名称、典型用法、组合规则、常见坑**，并配 **纯文本行** 便于复制进语料。

## 3. 竞品与生态参考（方向性，非背书）

- **Schema.org** 文档与 Google Search Central 对 Article、FAQ、Speakable 的说明。
- **llms.txt** 社区讨论：作为「站点 README for LLMs」的补充，非搜索引擎唯一标准。
- **Common Crawl / 学术语料** 偏好：干净正文、稳定 URL、明确许可边界。

## 4. 交付给产品经理的要点

1. 在现有 Astro 多站点架构上，增设 **「语料 / 参考」频道**，与主资讯频道隔离，避免干扰原有编辑节奏。  
2. 每篇语料文固定 **机器可抽取块**（TL;DR、实体列表、结构化小节、frontmatter FAQ）。  
3. 工程侧提供 **`/llms.txt`、纯文本镜像、（可选）Speakable**，运营按模板批量供稿。

---

**研究结论**：SEO 与「AI 友好」的交集是 **结构化、可引用、诚实边界**；额外增加 **站点级 llms 说明 + 纯文本镜像 + 强化 JSON-LD** 可提升被各类自动化管线消费的概率。
