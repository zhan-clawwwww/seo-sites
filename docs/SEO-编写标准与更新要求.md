# SEO 内容编写标准与日常更新要求（WordOK / seo-sites）

> **权威说明**：自 **2026-04** 起，**日常内容更新**以本文档为准。仓库内其他文档（如 `article-update-rules.md`）若与本篇在**更新范围、篇数**上冲突，**以本文为准**；工作流细节可与之互补。

---

## 1. 目标与原则

| 目标 | 说明 |
|------|------|
| **搜索友好** | 标题、摘要、结构、时效与深度信号对齐 Google **Helpful Content** 方向，避免堆砌与低质灌水。 |
| **读者有用** | 事实锚点 + 可解释的分析框架 + 可检验的前瞻（含证伪条件），而非转载摘要拼盘。 |
| **站点一致** | 全站已发布文章 **正文语言为英文**；元数据、分类、标签与 `sites/{站点}/config.json` 约定一致。 |

---

## 2. 日常维护范围与产量（必守）

### 2.1 活跃专栏（仅以下四个）

仅对下列目录产出**新文章**，不再把其他站点目录作为日常更新对象：

| 站点目录 | 主题定位（面向英文检索） |
|----------|-------------------------|
| `sites/ai/` | AI、LLM、企业与产业应用、智能体与工具链等 |
| `sites/web3/` | Web3、加密与链上生态、稳定币与合规、RWA 等 |
| `sites/tesla/` | Tesla、电动车、FSD/能源与交付与产业叙事等 |
| `sites/apple/` | Apple 软硬件、系统、服务与生态等 |

> 说明：仓库中若仍有 `vpn-usa`、`streaming`、`site-a`、`openclaw` 等站点，**日常 SEO 更新任务不包含**；仅在上级明确要求或全站专题时再动。

### 2.2 每日产量

- **每个活跃专栏：每个自然日 3 篇**新文章。  
- **合计：每天最多 12 篇**（4 × 3）。  
- 每篇**独立成稿**，禁止同一篇改标题敷衍多个专栏或多篇。

### 2.3 落盘路径与文件命名

```
sites/{ai|web3|tesla|apple}/posts/{站点slug}-YYYY-MM-DD-{英文短slug}.md
```

示例：

- `sites/ai/posts/ai-2026-04-27-agent-evaluation-observability-production.md`
- `sites/web3/posts/web3-2026-04-27-stablecoin-regulation-cross-border-payments.md`

- `pubDate` 与文件名中的日期**一致**，且为**真实发布日**。  
- 发布后执行 `git push`，由现有流程更新站点与 sitemap（见根目录 `README.md`）。

## 3. 语言与篇幅

| 项目 | 要求 |
|------|------|
| **正文语言** | **English only**（标题、正文、摘要、category、tags 均为英文）。 |
| **长度** | 正文 **不少于 3000 个英文单词**（统计时以 **frontmatter 之后 body 为准**，勿把 YAML 算入）。可用 `scripts/wordcount-post.mjs` 或编辑器统计。 |
| **信息密度** | 禁止同义反复、复制同一事实凑字数；每隔数段应有**新信息点**（数据、机制、对比、预测或边界条件）。 |

---

## 4. Frontmatter（与现站一致）

每篇必须包含：

```yaml
---
title: "English title: domain keywords + time window or angle (natural, not clickbait)"
pubDate: YYYY-MM-DD
author: "WordOK Tech Publications"
category: "English category aligned with the site column"  # e.g. Artificial Intelligence, Web3, Electric Vehicles, Apple
tags: ["5-10 concrete entities or long-tail phrases", "include 2026 when relevant"]
excerpt: "1-2 sentences: news/freshness hook + why the analysis/forecast matters"
---
```

- **title**：含领域核心词 + 时间或场景，避免纯标题党。  
- **excerpt**：独立撰写，**不要**简单复制首段；应概括「今日锚点 + 阅读价值」。  
- **tags**：具体实体与长尾词为主，避免无意义泛词堆砌。

---

## 5. 正文结构（SEO + 可读性）

以下块为**推荐骨架**，顺序可微调，但**「事实层 + 前瞻层」**二者不可缺一。

1. **开篇**  
   - 发布日、语言说明；可选：读者对象、免责声明（金融/安全/非投资建议等按主题需要）。  
   - 用一段说清：**为什么现在值得读**、本文回答什么问题。

2. **近期事实/行业锚点**  
   - 与主题强相关的 **近 7～14 天**公共信息（媒体报道、公告、财报电话会、发布说明等），**转述归纳**，避免大段抄袭。  
   - 关键事实宜有**多源交叉**意识；若报道矛盾，写明「不同信源差异」而非编造统一结论。

3. **分析主体**  
   - 分节使用 **H2 / H3**，层级清晰。  
   - 列表、表格仅用于**真的有助于理解**的归纳，不用于灌水。

4. **前瞻与可证伪性（非承诺）**  
   - 明确标注为 *forecast / scenario*，非既定事实。  
   - 至少两个时间尺度：**0–3 个月**、**3–12 个月**（或按主题调整，但须有两个不同尺度）。  
   - 重要预测应带 **falsifier**（何种公开信号出现则表明该判断不成立）。  

5. **对读者的行动含义**  
   - 可执行：检查清单、决策参考、或按角色（投资者/从业者/普通用户）分工的「应关注什么」。  

6. **风险、误区与边界**  
   - 涉及 **YMYL**（金融、安全、健康等）时，区分 **事实 / 行业惯例 / 作者推断**，避免绝对化投资建议或医疗结论。  

7. **结尾**  
   - 小结 + 自然收束；可含 3～6 个与专栏相关的长尾表达，**融入句子**，禁止关键词无意义重复列表。

## 6. SEO 与 E-E-A-T 要点（技术项）

- **时效**：`pubDate`、文内日期与「近周/本月」叙述一致，避免混用旧闻冒充新文。  
- **结构化数据**：以站点实现为准；作者侧保证 **唯一 H1**（与 title 可一致或略调）、逻辑标题层级。  
- **E-E-A-T**：专业主题写清信息边界；署名 `author` 与站点品牌一致。  
- **内链**：若同站已有相关旧文，在正文中**自然**增加指向（同一 `sites/{slug}` 下优先）。  
- **金融/加密类**：固定包含非投资建议 disclosure（如 *not financial advice*），**不提供**具体「买卖价目」作噱头。

---

## 7. 禁止项（红线）

- 中文作为**已发布正文**（元数据若工程需要中文，以仓库既有约定为准，但文章页面主体须为英文）。  
- 低于 **3000 英文词** 的「凑数文」、明显 AI 车轱辘话重复。  
- 关键词 **堆砌**（同义词机械重复、文末关键词列表）。  
- 虚构不存在的**具体数据、日期、引用**；不确定处用行业通用表述并标明不确定性。  
- 同一内容换标题多发于多专栏。

## 8. 发布前自检清单（每篇必过）

- [ ] 路径与命名符合 `sites/{ai|web3|tesla|apple}/posts/`，`pubDate` 与文件名日期一致。  
- [ ] 英文正文 **≥ 3000 words**（body），无恶意灌水。  
- [ ] Frontmatter 五项齐全且与专栏主题一致。  
- [ ] 含**近期锚点** + **前瞻/场景**，且前瞻带**假设或证伪**表述。  
- [ ] 金融/安全类 disclaimer 与边界已写（如适用）。  
- [ ] 已 `git add` / `commit` / `push`，构建与 sitemap 可按现有说明预期更新。
